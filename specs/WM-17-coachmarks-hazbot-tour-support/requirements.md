# Coachmarks: Hazbot Tour Support

**Jira**: https://concord-consortium.atlassian.net/browse/WM-17 (shared with the wildfire-model renderer story; no separate coachmarks ticket)
**Repo**: https://github.com/concord-consortium/coachmarks
**Driver spec**: wildfire-model `specs/WM-17-hazbot-visual-feedback-overlay-renderer/requirements.md` (the consumer that motivates these changes)
**Implementation Spec**: [implementation.md](implementation.md)
**Status**: **In Development**

## Overview

Three additions to `@concord-consortium/coachmarks` needed by wildfire-model's Hazbot visual-feedback overlay renderer (WM-17): (1) **action-gated tour advancement** with lazy per-step anchoring, (2) a **Hazbot-theme robot avatar badge** gated to tour popovers, and (3) **popover image/figure support**. All three are library-repo changes; the wildfire renderer, the `(ruleSetId, categoryId)` map, the extractor, and the authored tours stay in WM-17 (wildfire-model), which consumes a published coachmarks version.

## Project Owner Overview

The wildfire Hazbot coach needs to walk a student through a short sequence of actions ("Restart, then click Setup, then adjust the panel"), pointing at each control as the student reaches it. Today the coachmarks library can only show a fixed set of pre-located steps advanced by a Next button — it can't wait for the student to act, and it can't point at a control that doesn't exist on screen yet (like a panel that only opens after a click). This story adds that "advance when the student does the step" capability to the shared library, plus two smaller visual pieces the Hazbot design calls for (the robot avatar on the bubble, and the ability to show a picture in a bubble). Because coachmarks is a shared library (also used by the CODAP theme), these are built and reviewed as general library features.

## Background

Current library surface (verified):
- `createCoachmarksEngine(opts)` → `EngineHandle` with `drive(steps)`, `highlight(step)`, `moveNext()`, `movePrevious()`, `moveTo(i)`, `refresh()`, `destroy()`, `dismissPopover(i)` ([src/types.ts](../../src/types.ts), [src/engine.tsx](../../src/engine.tsx)).
- Step model: `EngineStep = PopoverSpec | PopoverGroup`; `PopoverSpec = AnchoredPopover | ViewportPopover`. `AnchoredPopover.element` is a live `HTMLElement` (with optional `ringElement`); `ViewportPopover` is anchorless (positioned by `position`).
- Options: `showButtons`, `disableButtons`, `showProgress` + `progressText`, `showOutlineRing`, `arrow`, `parseMarkdown`, `closeIcon`, `initialFocus`, etc.
- Themes: `styles/base.css` + `styles/hazbot.css` + `styles/codap.css`. The hazbot theme ([src/styles/hazbot.css](../../src/styles/hazbot.css)) has **no** robot avatar element today.

Key constraint motivating change #1: anchors are resolved by the consumer **up front** and passed as live elements. A step whose primary element is set but **not laid out** fires `onCancelRequested` (the engine requests cancel; the consumer then tears the tour down via `destroy()`) ([src/engine.tsx](../../src/engine.tsx) ~L135-138); passing `element: undefined` instead degrades the step to an anchorless viewport popover. So a tour whose later steps target controls that only appear after earlier steps' actions (Hazbot's "Restart → click Setup → the now-open Setup panel") cannot be expressed as a static `drive([...])`.

## Requirements

### 1. Action-gated tour advancement (headline; largest change)

- A driven tour can advance in response to the **student performing the step's action** (e.g. activating the anchored control, or an app-observable state change), not only via the Next button.
- **Lazy per-step anchoring**: a step's target is resolved/measured when that step becomes active, so a step may target an element that did not exist when the tour started. If the target is not yet present when the step is entered, the engine **waits for it** (polls/observes) rather than cancelling.
- **Deferred target representation**: because a lazy/gated step's anchor may not exist at tour-construction time, such a step declares its target as a **CSS selector string** (see resolved Open Question) that the engine resolves *at activation* (and waits for via MutationObserver if absent), not as a pre-resolved live `HTMLElement`. This is a new step-target form distinct from today's `AnchoredPopover.element: HTMLElement` (live element) and `ViewportPopover` (anchorless). A gated tour may use selector targets uniformly for every step (present or deferred), since the engine resolves-and-waits either way. The selector resolves to the first match (`querySelector`), so the consumer ensures uniqueness (`data-testid` already does). Existing live-element steps keep working unchanged.
- The consumer declares, per step, how the step is satisfied (the "advance trigger"), via **two complementary paths** (see resolved Open Question):
  - **Declarative** `advanceOn: { event: "click" }` (a DOM event on the step's anchor): the library attaches the listener to the anchor and cleans it up on step exit. Covers anchor-activation steps (e.g. "Restart your model", "click the Setup button").
  - **Imperative** `moveNext()`: the consumer detects an app-observable state change through its own means and advances. Covers steps with no anchor DOM event (e.g. "place a spark in each zone" on the Three.js canvas, "change the zones then run the model again" signalled by the consumer's existing run/state events).
  - **No polled predicate**: the library does not poll a `() => boolean`; app-state gating goes through the imperative path so the library stays free of timers/intervals. Exact field names in `implementation.md`.
- **Waiting is unbounded and cancellable, with no orphan bubble**: when a gated step's target is not yet laid out, the engine does **not** advance into an unanchored step. It holds on the current (already-anchored, already-dismissable) step until the target becomes laid out, then advances and anchors. The action-gated advance is simply **deferred** until the target appears (wait-for-target is a property of gated/selector steps; see Back-compat). Because a dismissable popover stays on screen throughout, the wait is bounded only by the student: there is no timeout, and the student can cancel at any time (close button / Escape) or the consumer can `destroy()` the handle. Cancelling fires `onCancelRequested` as usual. (The first step is expected to have a present target at `drive()` time.)
- **Back-compat**: existing Next-button / imperative tours are unaffected when no advance trigger is supplied. Specifically, wait-for-target (waiting for an absent anchor instead of cancelling) applies **only** to steps that opt into gating/lazy anchoring; a step that passes a pre-resolved live `HTMLElement` retains today's cancel-on-not-laid-out behavior. Note today's behavior has **two** paths and the refactor must preserve both: the **first** step of a `drive()`/`highlight()` cancels via the `startStepWithDeferral` rAF guard ([src/engine.tsx:130-139](../../src/engine.tsx#L130-L139)), while a **mid-tour** step reached by `moveNext()`/`moveTo()` enters directly ([src/engine.tsx:320-321](../../src/engine.tsx#L320-L321)) and cancels via the target-watcher's microtask ([src/use-target-watcher.ts:17-29](../../src/use-target-watcher.ts#L17-L29)) → `onCancelRequested`. The unifying `goToStep()` refactor (see `implementation.md`) routes mid-tour advances through the deferral path too, so it must guarantee a not-laid-out live step still fires `onCancelRequested` **exactly once** (the deferral defers `enterStep`, so the popover never mounts and the target-watcher never double-fires). The wait-for-target behavior is never applied globally.
- Cancellation semantics remain available (close button / Escape / explicit cancel); "target absent" is no longer an automatic cancel for gated steps.
- **Gated degrade-on-removal (anchor removed mid-tour does not cancel)**: in an `actionGated` tour, when a step's primary anchor leaves layout (DOM removal or `display:none` / zero-size — the same predicate `useTargetWatcher` watches), the engine **re-renders that step as an anchorless centered popover** — keeping its title/description/image, progress text, and Done/close buttons — instead of firing the anchor-removal `onCancelRequested`. One rule, two cases: **(a) a held step during a pending wait** whose anchor is removed (a consumer's wizard "Next" button unmounts as the next sub-panel mounts) re-floats and then advances when the next target appears (this subsumes the Round-4 held-during-wait suppression); **(b) a step with no pending advance** — typically the **terminal** Done step anchored to a transient container — whose anchor is removed (the consumer closes that container to complete the flow) re-floats and stays until Done/close. A re-floated step keeps its identity and step number (it does **not** advance or increment progress; it is the same step, now anchorless). Only the close button / Escape / explicit `destroy()` cancel a gated tour; anchor removal never does. **Non-gated tours are unchanged** — anchor removal still cancels via `onCancelRequested` (the existing `useTargetWatcher` path). Motivated by the wildfire driver: 24's "Click Next" anchor is removed mid-wait, and every Zeplin-anchored terminal "Setup panel"/"Wind" step's anchor is removed when its "…then run the model again" instruction closes the panel.
- **Gated steps hide passive navigation; only the terminal done button remains**: a gated step shows **no Next button and no Previous button** (advancement is the student's action / imperative `moveNext()`, never a passive button), matching the Zeplin design (avatar + title/body + progress text + close + arrow only). To match, the keyboard nav paths are suppressed on gated steps: ArrowRight must not skip the gate and ArrowLeft must not retreat (today both fire `moveNext()`/`movePrevious()` at [src/use-keyboard-control.ts](../../src/use-keyboard-control.ts#L70-L75) regardless of button visibility). The tour's **terminal step still shows its done button** (`[Got it!]`). Because the engine-level `showButtons` cannot express "no Next on intermediate steps but a done button on the last" (the done button is the Next button rendered as Done at [src/popover.tsx:720-732](../../src/popover.tsx#L720-L732), both gated by `showButtons.includes("next")`), this requires gated-aware (or per-step) button rendering, not a single `showButtons` value. The **close button and progress text remain** on every step. Concretely, a gated tour that ends in a Done button must still include `"next"` in `showButtons` (e.g. `["next", "close"]`): the gated rule suppresses Next on intermediate steps and renders it as Done only on the terminal step. Omitting `"next"` hides the terminal Done too, so `["close"]` is correct only for a tour whose final step is itself action-gated.
- **A gated tour does not steal focus on advance**: in an `actionGated` tour, every advance follows a student action in the app (an `advanceOn` click on the anchor, or the consumer calling `moveNext()` after an app-observable change such as spark placement / re-run), so the engine must **not** move keyboard focus into the popover on step entry; focus stays where the student's action left it (the app control they just operated), so keyboard/SR users are not pulled out of their workflow. This overrides today's per-step-entry initial-focus pull ([src/popover.tsx:473,481-490](../../src/popover.tsx#L481-L490)) for gated tours, applied uniformly (the engine cannot, and need not, distinguish an `advanceOn` auto-advance from an imperative `moveNext()` advance: both follow a student action). **Non-gated** tours are unaffected and keep today's focus-to-popover behavior on button/imperative advance. **Scope**: this no-focus-steal governs the **initial-focus** effect only. The separate `pullFocusFromIframe` rescue ([src/popover.tsx:462-469](../../src/popover.tsx#L462-L469)) — which pulls focus to the popover when an `IFRAME` holds it on step entry — stays intentionally active in gated tours (otherwise a keypress is lost into the cross-origin iframe); a gated consumer that does not want it sets `pullFocusFromIframe: false`. WM-17 never triggers it (wildfire's anchors are DOM buttons and the map is a `<canvas>`, never an iframe).
- **Terminal gated step is the focus exception (only when it shows Done)**: a **Done-terminated** final gated step (the one rendering the `[Got it!]` Done button) **does** pull focus to the popover (or directly to the Done button) on entry, because there is no further in-app action expected and the Done button is the sole completion affordance: a keyboard/SR user must be able to find and activate it without blind-Tabbing into the end-of-`<body>` portal. **Intermediate** gated steps keep no-focus-steal — and so does a final step that is *itself action-gated* (the `["close"]` case below: no Done, completes via an in-app action), since there a further action is still expected. The exception is therefore keyed on Done-terminated, not position: the focus mode is `actionGated && !(isLast && showNext) ? "none" : (opts.initialFocus ?? "popover")` (not a flat `actionGated ? "none"`, nor `isLast` alone). All wildfire tours are Done-terminated (driver spec), so they always take the exception on their final step.
- **Lifecycle callbacks fire normally under gated advance**: `onHighlightStarted` fires once per gated step (with the correct `activeIndex`), and `onDeselected`/`onDestroyed`/`onCancelRequested` fire as today, so a consumer can anchor and log each step and detect completion/dismissal without library-side logging.

### 2. Hazbot-theme robot avatar badge (gated to tours)

- Render the Hazbot robot avatar badge (red circular robot face overlapping the popover's top-left), per the Hazbot design screenshot.
- **Default-on, explicitly suppressible**: the avatar renders by default on hazbot-theme popovers (tours and `highlight()` alike). A consumer can suppress it via an explicit option (default = shown). The option must be settable at a granularity that lets a single consumer show the avatar on some popovers and not others, so wildfire can suppress it on its intro "Show me" `highlight()` bubble (which already points *at* the robot button, making a badge there redundant) while leaving it on for the launched tour. Whether that is a per-call/per-popover override or achieved by using separate engine instances is an `implementation.md` detail (see the driver spec: the intro `highlight()` and the tour `drive()` may run on one engine).
- Artwork lives in the library (the avatar is a coachmarks/hazbot-theme asset, consistent with the theme owning its visuals).
- **Decorative**: the avatar is purely decorative (the speaker identity is already carried by the popover text and theme), so it must be hidden from assistive technology (`aria-hidden` / empty `alt`), consistent with the theme's existing arrow-SVG treatment ([src/popover.tsx:250-251](../../src/popover.tsx#L250-L251)). It must not add to the dialog's accessible name/description.

### 3. Popover image/figure support

- A popover (anchored or viewport) can render an **image/figure** in addition to `title`/`description`.
- API: an `image?: ReactNode` field on the popover spec (see resolved Open Question), rendered in a dedicated figure slot with a sensible default position relative to title/description (exact placement an implementation detail). The consumer owns the element's `alt`/`aria`: for the meaningful mountain imagery the `arrowText` description already supplies a text equivalent, so the SVG may be `aria-hidden` with the description carrying the meaning, or carry its own `role="img"`/`aria-label`.
- Consumed by wildfire ruleset 25 Cat 4 (mountain top/bottom imagery); wildfire passes a placeholder SVG initially and the final art later, with no library change.

  2. **Then yalc-link to wildfire to wire it up and validate integration** (build → `yalc push` → wildfire picks up; bump version per iteration; `~/tmp/wildfire-yalc.md`).   3. **Then `npm publish`** (by the user) the version WM-17 pins.
  - Tests: vitest (`npm test`); lint/types: `npm run check`.

- Action-gating likely builds on the existing `moveNext()`/`moveTo()` imperative API plus the step-entry path in [src/engine.tsx](../../src/engine.tsx) (where `isLaidOut`/`onCancelRequested` currently live) and `use-target-watcher.ts` (which already observes target presence/removal — a natural hook for "wait for target").

- Keep changes theme-agnostic where possible (image support and advancement are core features; only the badge is hazbot-specific).
- Ship as a published version (`npm publish`, by the user) that WM-17 pins (step 3 of the workflow above).

## Open Questions

### RESOLVED: Advance-trigger API shape
**Context**: How does a consumer declare when a gated step is satisfied?
**Options considered**:
- A) Declarative per-step `advanceOn`: `{ event: "click", target?: "anchor" }` and/or a predicate `() => boolean` the engine polls/observes.
- B) Imperative-only: consumer wires its own listeners and calls `moveNext()`; the library only adds lazy anchoring + wait-for-target.
- C) Both: declarative for common cases (click the anchor), imperative escape hatch for complex ones.

**Decision**: **C, with the declarative form limited to a DOM event on the anchor (no polled predicate).** Verified against the authored tours: advance actions split into anchor activations (Restart/Setup clicks) and app-observable state changes (spark placement on the raycast-driven Three.js canvas, zone config + re-run). These have different natural owners: the library owns the anchor element (declarative `advanceOn: { event: "click" }`, library-managed listener), the consumer owns app state (imperative `moveNext()` off its existing event stream). Dropping the polled `() => boolean` keeps the library timer-free. Worked example (ruleset 23 Cat 2): steps 1-2 use `advanceOn: { event: "click" }` on the Restart/Setup anchors; step 3 ("configure zones + run again") is the **terminal Done step**, completed by the `[Got it!]` button rather than an advance. Note: across the authored wildfire tours every app-observable action (spark placement, zone config + re-run) lands on the terminal Done step, so wildfire never actually calls `moveNext()` — the imperative path is retained for generality and is exercised by the demo's gated-tour section.

### RESOLVED: Deferred target representation for lazy/gated steps
**Context**: A lazy/gated step's anchor may not exist when the tour is constructed, so its target cannot be a pre-resolved `HTMLElement`. How is the target declared so the engine can resolve it at step activation? (Distinct from the advance-trigger question above, which governs *when* the step is satisfied, not *what* it anchors to.)
**Options considered**:
- A) CSS selector string (engine queries the document when the step becomes active).
- B) Resolver function `() => HTMLElement | null` (consumer owns lookup; engine polls/observes until non-null).
- C) Both (selector for the common case, resolver escape hatch).

**Decision**: **A — CSS selector string.** One-to-one fit with wildfire's existing `data-testid` anchor model (driver spec: targets are `data-testid` strings resolved via `document.querySelector` at open time), and it composes with the existing removal-watching MutationObserver ([src/use-target-watcher.ts](../../src/use-target-watcher.ts)) for poll-free wait-for-appearance. A resolver function is unnecessary: nothing in the 11 rulesets needs a computed target without a stable `data-testid`. A gated tour may use selectors uniformly for all steps; `querySelector` first-match means the consumer ensures uniqueness.

### RESOLVED: Badge gating mechanism
**Context**: Control whether the robot avatar shows on a given popover.
**Options considered**:
- A) Engine option (e.g. `showAvatar: boolean` / `avatar?: ReactNode`).
- B) CSS modifier class the consumer toggles on the popover root.
- C) Theme variant.
- D) Reuse the engine's existing `kind: "tour" | "highlight"` distinction (already gates progress text at [src/popover.tsx:696](../../src/popover.tsx#L696)) so the badge auto-shows on tours only.

**Decision**: **Explicit show/hide option, default = shown** (a refinement of A, default-on instead of default-off). The avatar renders by default on hazbot-theme popovers and is suppressed by an explicit flag. Option D was considered but rejected: tying the badge to `kind === "tour"` cannot suppress it on a specific tour, and the intro/tour may share one engine. The flag must be settable so the intro popover can differ from the tour; per-call/per-popover override vs. two engine instances is an `implementation.md` detail.

### RESOLVED: Image API
**Context**: How is the image supplied and laid out?
**Options considered**:
- A) `image?: string` (URL/data-URI) on the popover spec, fixed layout above the description, required `alt`.
- B) `image?: ReactNode` (consumer supplies the element), more flexible, consumer owns layout/alt.
- C) A generic `media`/`figure` slot supporting either.

**Decision**: **B — `image?: ReactNode`.** Fits the consumers' SVG-as-React-component pipeline directly (wildfire imports `*.svg` as components and renders `<MountainPlaceholder />` → `<MountainFinal />` with no library change), simplest surface, consumer owns the element's `alt`/`aria`. A URL-string form (A) is unnecessary: both relevant consumers are React (wildfire, and CODAP v3, now the default, is React), so there is no non-React consumer needing a plain URL.

### RESOLVED: Does a gated tour need working "Previous"/back?
**Context**: Action-gated forward advance is required; backward navigation through already-performed steps may or may not be meaningful (the app state has moved on).
**Options considered**:
- A) Forward-only for gated tours (hide/disable Previous).
- B) Support Previous, re-anchoring to whatever is currently present.

**Decision**: **A — forward-only.** Matches the driver spec's stated model (the walk-throughs are forward action sequences advanced only by the student's action, "not via a passive Next button"); backward nav is incoherent once the action has mutated app state. The Zeplin design (intermediate "Step 2 of 3" coach mark) shows **no Next and no Previous button** at all, only avatar + text + progress + close + arrow, so gated steps hide *both* nav buttons; the terminal step keeps its `[Got it!]` done button. Forward-only/no-next is not expressible via `showButtons` alone (the done button is the Next button; ArrowLeft/ArrowRight also navigate via the keyboard), so the library adds gated-aware button + keyboard suppression. Captured as a requirement bullet under #1.

## Self-Review

### Senior Engineer / API Designer

#### RESOLVED: Lazy anchoring is incompatible with the current `element: HTMLElement` model, and the requirements never name the deferred-target representation
**Resolution**: Added a "Deferred target representation" requirement under #1 and a parallel OPEN question capturing the selector-vs-resolver choice.
Requirement #1 bullet 2 ("a step may target an element that did not exist when the tour started") cannot be expressed with today's step model. `AnchoredPopover.element` is a **required live `HTMLElement`** ([src/types.ts:4](../../src/types.ts#L4)); the only anchorless form is `ViewportPopover` (`element?: undefined`). There is no selector/resolver/lazy form. So the requirements imply a new target representation (CSS selector, `() => HTMLElement | null` resolver, or similar) that the engine resolves at step-entry time, but no requirement states this. Without it, "lazy per-step anchoring" has no concrete surface to attach to and the implementation spec has nothing to anchor against. Suggested resolution: add a requirement that a gated/lazy step declares its target as something resolvable at activation (not a pre-resolved element), and capture the representation choice as an OPEN question alongside the advance-trigger one.

#### RESOLVED: Is lazy anchoring opt-in (tied to an advance trigger) or always-on? Back-compat with the current cancel-on-not-laid-out path is ambiguous
**Resolution**: Tightened requirement #1 back-compat bullet to scope wait-for-target to opt-in gated/lazy steps only; live-element steps keep today's cancel-on-not-laid-out behavior.
Today a step whose primary element is set but not laid out cancels via `onCancelRequested` after one rAF ([src/engine.tsx:130-139](../../src/engine.tsx#L130-L139)). Requirement #1 bullet 2 ("waits for it ... rather than cancelling") reads as a general behavior change, but bullet 4 scopes back-compat to "when no advance trigger is supplied." These conflict: if lazy/wait-for-target applies to all steps, it silently changes existing tours' cancel behavior (a behavior break for current consumers, including CODAP); if it applies only to gated steps, the requirements should say the wait-for-target behavior is gated on the same opt-in as advancement. Suggested resolution: state explicitly that wait-for-target replaces cancel-on-absent **only** for steps that opt into gating, and that non-gated steps retain today's cancel-on-not-laid-out semantics.

#### RESOLVED: Background wording "tears down the tour" overstates what the engine does
Background said a not-laid-out step "fires `onCancelRequested` and tears down the tour." The engine only **fires the callback** ([src/engine.tsx:138](../../src/engine.tsx#L138)); teardown (`destroy()`) is the consumer's responsibility (per the callback doc [src/types.ts:145-147](../../src/types.ts#L145-L147); wildfire wires it at [hazbot-button.tsx:108](../../../wildfire-model/src/components/hazbot-button.tsx#L108)). **Resolution**: reworded to "fires `onCancelRequested` (the engine requests cancel; the consumer then tears the tour down via `destroy()`)."

### QA Engineer

#### RESOLVED: No failure/timeout semantics for "waits for it" — what happens when the target never appears or the student never performs the action?
**Resolution**: Waiting is unbounded (no timeout). The engine holds on the current, already-dismissable step (Next disabled for button-driven advance, auto-advance deferred for action-gated) rather than entering an unanchored step, so a cancellable popover is always on screen; the student cancels via close/Escape or the consumer via `destroy()`. Added as a requirement bullet under #1.

### Product Manager / Library Integrator

#### RESOLVED: Badge-gating open question omits the engine's existing `kind: "tour" | "highlight"` distinction
The store already distinguishes `kind: "tour"` vs `"highlight"` and gates progress text on it ([src/popover.tsx:696](../../src/popover.tsx#L696)), so reusing `kind` was a candidate gating mechanism. **Resolution**: added as option D to the badge-gating question, then rejected in favor of an explicit show/hide option defaulting to shown (kind-based gating can't suppress the badge on a specific tour, and the intro/tour may share one engine). Requirement #2's gating bullet was rewritten to the default-on/opt-out model.

### WCAG Accessibility Expert

#### RESOLVED: The robot avatar badge has no specified accessible treatment (decorative vs. labeled)
The popover root is `role="dialog"` with wired `aria-labelledby`/`aria-describedby` ([src/popover.tsx:660-666](../../src/popover.tsx#L660-L666)); an unlabeled image would add SR noise, and the codebase already hides decorative SVGs (the arrow at [src/popover.tsx:250-251](../../src/popover.tsx#L250-L251)). **Resolution**: added a requirement bullet under #2 declaring the avatar decorative and hidden from assistive tech.

#### RESOLVED: Action-gated advance vs. focus management is unspecified — auto-advance may yank keyboard focus away from the control the student is operating
Today the engine pulls focus into the popover on every step entry ([src/popover.tsx:473,481-490](../../src/popover.tsx#L481-L490)) with no way to distinguish "user clicked our Next button" from "tour auto-advanced because the student acted in the app." **Resolution**: added a requirement under #1 that action-gated (auto) advances do not steal focus (focus stays where the student's action left it); button-driven/imperative advances keep today's behavior.

**Resolution**: Added a single-in-flight-wait guard to `goToStep()` ([implementation.md](implementation.md)): a new advance no-ops while a wait is pending (`waiting`/`waitDispose` set), since the pending `next` is unchanged — at most one `waitForTarget` is live and exactly one transition runs when the target appears. The `goToStep()` plan for an absent selector target ([implementation.md](implementation.md), "Selector target resolution"): "register the current `seqId`, call `waitForTarget(primary.target, …)`, and **do not leave the current step**. On resolve (guarded by `seqId`/`destroyed`), `leaveStep()` + `enterStep(next)`." Verified gap: while waiting, the engine stays on the prior step, so that step's popover stays mounted and its `advanceOn` listener stays attached (`useAdvanceTrigger` keys on the active primary). Nothing guards a **second** `moveNext()` during the wait window — a re-click of the still-live `advanceOn` anchor (e.g. clicking "Setup" twice) or a second imperative `moveNext()` (e.g. two rapid `SimulationStarted` events). ## Self-Review — Round 4 (wildfire-renderer contract verification, 2026-06-20)

Surfaced while writing the wildfire-model WM-17 implementation plan against this published API. Each finding was deep-dive-verified against both the coachmarks source and the wildfire source before being written, per the driver story's CRITICAL RULE: any wildfire need that cannot be cleanly expressed against the coachmarks API as specced is recorded here for a decision rather than worked around on the wildfire side.

### Senior Engineer / API Designer

#### RESOLVED: A held step's anchor removal during wait-for-target cancels the tour — a step whose own advance removes its anchor (wizard "Next" → "Wind") is unexpressible
**Resolution: A — suppress the held step's removal-cancel once an advance is in flight.** While `goToStep()` is waiting for the next step's target (`waitDispose` non-null), the engine is committed to leaving the current step, so the held step's `useTargetWatcher` removal-cancel is ignored for the duration of the pending wait; the tour still tears down normally on student cancel (close/Escape) or `destroy()`, and advances when the next target appears. Folded into the coachmarks `implementation.md` "Selector target resolution" step (the wait-for-target / re-entrancy-guard block) with a regression test (advance that removes the held anchor while the next target is briefly absent advances to the next step instead of cancelling). This makes "advance removes the held anchor" first-class for any wizard-like consumer; the wildfire plan pins the published version that includes it.

**Update (cross-repo review, 2026-06-20): generalized into "gated degrade-on-removal" to also cover the Zeplin-anchored terminal-on-panel-close case.** A second wildfire interaction surfaced with the *same* shape: the terminal "Setup panel"/"Wind" steps are **anchored to the Setup panel** per the Zeplin design ("Step N of N", arrow at the panel), and their "…then run the model again" instruction closes (unmounts) the panel — removing the terminal step's anchor with no advance pending, which `useTargetWatcher` would cancel (losing `[Got it!]`, mis-logging a dismissal). Rather than carry this `waitDispose`-gated suppression *plus* a separate terminal rule, `pre.8` ships a single **gated degrade-on-removal** behavior: **in an `actionGated` tour, a step whose anchor leaves layout re-floats as an anchorless centered popover (keeping its content, step number, and Done/close buttons) instead of firing the anchor-removal cancel.** The Round-4 held-during-wait case is a sub-case — the held step re-floats briefly and then advances when the next target appears; the terminal case re-floats and stays until `[Got it!]`. This is on the WM-17 critical path (the common Zeplin-anchored terminals across 23/32/33/34/35/54, plus 24's Next). See requirement #1 and `implementation.md`. Below is the original finding for the decision log.


**The gap.** The wait-for-target hold ([implementation.md](implementation.md), "Selector target resolution": on an absent next-target the engine registers `waitForTarget(...)` and **"does not leave the current step"**) assumes the **held** (current) step's anchor stays present while waiting for the *next* step's target to appear. That holds for the canonical Hazbot case the spec designed around — "click the Setup button → wait for the Setup panel" — because the held anchor (the bottom-bar **Setup** button) persists in the DOM while the panel mounts. It does **not** hold for the wildfire **Setup-wizard internal** transition, where the action that advances the tour also **removes the held step's anchor**.

**Concrete failure (wildfire ruleset 24, the only Next→Wind tour).** Authored tour (verified in [wildfire-model/src/hazbot/rule-sets/24.ts](../../../wildfire-model/src/hazbot/rule-sets/24.ts), cats 2-4): step 3 = "Click the Next button" (anchored to the Setup-panel **Next** button, `advanceOn: {event:"click"}`); step 4 = "Change the Wind Direction and Wind Speed. Then run the model again." (anchored to the **Wind** section). In the Setup wizard ([wildfire-model/src/components/terrain-panel.tsx](../../../wildfire-model/src/components/terrain-panel.tsx)) the Next button lives in the `currentPanel === 1` sub-panel (terrain-panel.tsx:318) and the Wind section lives in the `currentPanel === 2` sub-panel (terrain-panel.tsx:331-339). Clicking Next runs `showNextPanel()` → `setCurrentPanel(2)` (terrain-panel.tsx:134-141), which **unmounts panel 1 (removing the Next button) and mounts panel 2 (revealing Wind)** in one React commit. So on that single click:
1. `advanceOn` fires → `moveNext()` → `goToStep(4)`. Step 4's target (`[data-testid="terrain-wind"]`) is **not yet laid out** (panel 2 hasn't committed) → the engine registers `waitForTarget` and **holds step 3**, fully mounted.
2. The same click's `setCurrentPanel(2)` commit **removes step 3's anchor** (the panel-1 Next button).
3. Step 3 is still mounted, so its primary popover's `useTargetWatcher` is still active. On the removal mutation it fires `onRemoved` → the engine's cancel path → `onCancelRequested` ([use-target-watcher.ts:30-34](../../src/use-target-watcher.ts#L30-L34) observes `document.body` for the held anchor leaving layout; verified the watcher cancels on removal of a mounted step's anchor). The wildfire consumer wires `onCancelRequested: () => engine.destroy()`, so **the tour dies** instead of advancing to the Wind step.

It is a **race**, not a deterministic cancel: step 4's appearance-watcher (`waitForTarget` for Wind) and step 3's removal-watcher (for Next) both observe `document.body` and fire on the **same** mutation batch (panel 1 out / panel 2 in commit together). If appearance wins, `leaveStep(3)` disposes step 3's removal-watcher before it fires and the tour advances; if removal wins, the tour cancels. Nondeterministic, environment-timing-dependent, and untestable to a stable outcome — unacceptable for an authored tour.

**Why it can't be cleanly worked around on the wildfire side.** The faithful expression of 24's intent (point at the Next button, advance when the student clicks it, then point at the now-revealed Wind section) requires holding the Next step across the panel swap, which is exactly what triggers the removal. Candidate wildfire workarounds all degrade or fail:
- *Anchor step 4 to the persistent panel container instead of the Wind section* — the container (`css.background`, terrain-panel.tsx:236) survives the panel swap, so step 4's target is present immediately, `goToStep(4)` does **not** wait, and `leaveStep(3)` runs before the removal matters (no race). But this abandons "outline the Wind section / point at the Setup panel" (24's vF) and anchors a generic bubble to the whole panel — a visible degradation of the authored design, and it still leaves the ring on an initially-absent Wind section.
- *Drop the separate "Click Next" step* — breaks 24's authored 4-step structure and the `arrowText` step-count invariant.
- *Observe `currentPanel` to drive imperative `moveNext()`* — `currentPanel` is component-local `useState` (terrain-panel.tsx:36), not host-observable; the host cannot see it.

None expresses the authored tour cleanly, so per the CRITICAL RULE this is recorded as a coachmarks-spec gap rather than silently degraded on the wildfire side.

**Suggested resolution (recommended: A).**
- **A) Suppress the held step's removal-cancel once an advance is in flight.** When `goToStep()` has requested an advance and is waiting for the next step's target (`waitDispose` non-null), the engine is already committed to leaving the current step, so the current step's anchor removal must **not** cancel the tour — disable/ignore the held step's `useTargetWatcher` removal-cancel for the duration of the pending wait (and still tear down normally once the next target appears or the student cancels via close/Escape). Small, general, and correct for any wizard-like consumer (a step whose advance disposes its own anchor); composes with the Round-3 single-in-flight-wait guard. This makes the "advance removes the held anchor" pattern first-class rather than a race.
- **B) Keep the prior step shown as an anchorless bubble if its anchor vanishes mid-wait** (instead of cancelling) — heavier; reverses the spec's deliberate "hold prior step, no orphan bubble" choice.
- **C) Document the limitation and require consumers to anchor the post-Next step to a persistent ancestor** (the wildfire workaround above) — no library change, but pushes a fragile, design-degrading constraint onto every wizard consumer and leaves the race latent for anyone who anchors to a real sub-panel control.

**Decision**: **A** (recorded above).
